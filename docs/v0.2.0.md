# Skill Inspector v0.2

## Background

Current implementation analyzes individual files as independent assets.

This does not match Hermes architecture.

Hermes organizes knowledge as Skill Packages:

The `skills/` **directory structure is as follows:**

skills/
├── category-name/          ← Category folder
│   ├── DESCRIPTION.md      ← Description of the category (optional)
│   └── skill-name/         ← Single skill folder
│       ├── SKILL.md        ← Required
│       ├── references/     ← Optional
│       ├── templates/      ← Optional
│       ├── scripts/        ← Optional
│       └── assets/         ← Optional


Note there is an exception: `skills/dogfood/` has no `DESCRIPTION.md` and no sub-skill folders; it is itself a single skill (containing `SKILL.md` + `references/` + `templates/`). Similarly, `skills/yuanbao/` and `skills/software-development/` also place skills directly rather than categories. Therefore, not all categories strictly follow the `DESCRIPTION.md` pattern—some skills are placed directly under the `skills/` root directory.

A skill package is the primary unit.
Individual files inside references/, templates/, scripts/, and assets/ belong to the package and should not be treated as standalone skills.

---

# Objective

Refactor Skill Inspector to become package-aware.

The primary analysis unit should be Skill Package.

Asset analysis becomes secondary.

# New Data Model

Category
↓
Skill Package
↓
Assets

---

# Category Model

Fields:

* name
* description (from DESCRIPTION.md if present)
* skill_count

---

# Skill Package Model

Fields:

* name

* category

* skill_content

* references[]

* templates[]

* scripts[]

* assets[]

* package_type

* confidence

---

# Asset Model

Fields:

* asset_type
* path
* content

Types:

* Reference
* Template
* Script
* Asset

---

# Package Classification

Classify the SKILL.md only.

Do NOT classify reference files independently.

Classify package into:

* Executable Skill
* Workflow
* Knowledge
* Preference
* Hybrid
* Unknown

Examples:

github-code-review
→ Executable Skill

developer-tool-promotion
→ Knowledge

content-publishing
→ Workflow

---

# Asset Inventory

For every package calculate:

references_count

templates_count

scripts_count

assets_count

github-code-review

References: 3

Scripts: 0
Assets: 0

# Package Complexity Score

Create a simple score.

Example:

complexity =
references_count

* scripts_count * 3
* assets_count

github-code-review

# Duplicate Detection

Run duplicate detection only on:

SKILL.md

Do not compare:

references/
templates/
scripts/

This currently creates many false positives.

# Success Criteria

The report should answer:

1. How many skill packages exist?

2. What kinds of skill packages exist?

3. Which categories dominate the skill library?

4. Which packages are most complex?

5. Which packages overlap?

6. How is knowledge organized inside Hermes?
